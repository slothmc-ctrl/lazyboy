# Internationalization (i18n)

## Overview

Sitegeist uses the mini-lit i18n system with TypeScript for type-safe translations. All user-facing strings must be internationalized to support multiple languages.

Currently supported languages:
- **English** (en) - Default
- **German** (de)

## Architecture

The i18n system is built on three layers:

1. **mini-lit** - Base i18n system with `i18n()` function
2. **pi-web-ui** - Web UI component translations
3. **Sitegeist** - Extension-specific translations

All three are merged together in [src/utils/i18n-extension.ts](../src/utils/i18n-extension.ts).

## How to Add New Translations

### Step 1: Add Type Declaration

First, declare your translation key in the `i18nMessages` interface in `src/utils/i18n-extension.ts`:

```typescript
declare module "@mariozechner/mini-lit" {
	interface i18nMessages {
		// ... existing keys ...
		"Your new key": string;
	}
}
```

### Step 2: Add English Translation

Add the English translation to the `lazyboyTranslations.en` object:

```typescript
const lazyboyTranslations = {
	en: {
		// ... existing translations ...
		"Your new key": "Your new key",
	},
	// ...
};
```

### Step 3: Add German Translation

Add the German translation to the `lazyboyTranslations.de` object:

```typescript
const lazyboyTranslations = {
	// ...
	de: {
		// ... existing translations ...
		"Your new key": "Ihr neuer Schlüssel",
	},
};
```

### Step 4: Use in Code

Import and use the `i18n()` function in your component:

```typescript
import { i18n } from "@mariozechner/mini-lit";
import "../utils/i18n-extension.js"; // Required for Sitegeist translations

// In your template:
html`<div>${i18n("Your new key")}</div>`
```

**IMPORTANT**: Always import `../utils/i18n-extension.js` in files that use Sitegeist-specific translation keys. This ensures the translations are loaded before use.

## Examples

### Simple String

```typescript
// Type declaration
interface i18nMessages {
	"Waiting...": string;
}

// Translations
en: { "Waiting...": "Waiting..." }
de: { "Waiting...": "Warte..." }

// Usage
html`<div>${i18n("Waiting...")}</div>`
```

### String Concatenation

Use template literals (NOT string concatenation):

```typescript
// ❌ BAD - String concatenation
i18n("Waiting for selection") + ": " + message

// ✅ GOOD - Template literal
`${i18n("Waiting for selection")}: ${message}`
```

### Parameterized Strings

For strings with dynamic values, use string interpolation OUTSIDE of i18n:

```typescript
// Type declaration
interface i18nMessages {
	"Found {count} items": string; // Note: This is just the key, not a function
}

// Translations
en: { "Found {count} items": "Found {count} items" }
de: { "Found {count} items": "{count} Elemente gefunden" }

// Usage - Replace placeholders manually
const count = 5;
const text = i18n("Found {count} items").replace("{count}", count.toString());
```

## Best Practices

### DO:

1. **Use descriptive keys** - Keys should clearly indicate the context
   ```typescript
   "Waiting for selection" // ✅ Clear
   "wait" // ❌ Too vague
   ```

2. **Keep keys in English** - The key itself should be the English text
   ```typescript
   "Open tabs": "Open tabs" // ✅ Key is English text
   "tabs.open": "Open tabs" // ❌ Don't use dot notation
   ```

3. **Import i18n-extension.js** - Required for Sitegeist keys
   ```typescript
   import "../utils/i18n-extension.js"; // ✅
   ```

4. **Use template literals for concatenation**
   ```typescript
   `${i18n("Prefix")}: ${value}` // ✅
   i18n("Prefix") + ": " + value // ❌
   ```

### DON'T:

1. **Don't hardcode user-facing text**
   ```typescript
   html`<div>Loading...</div>` // ❌
   html`<div>${i18n("Loading...")}</div>` // ✅
   ```

2. **Don't skip type declarations**
   - TypeScript will catch missing translations at compile time
   - If you see a type error, add the key to the interface

3. **Don't forget German translations**
   - Every English key must have a German equivalent
   - Use translation tools if needed, but verify accuracy

4. **Don't use i18n for:**
   - Internal logs: `console.log("Debug info")` ✅
   - API keys/URLs: `const url = "https://..."` ✅
   - Code constants: `const STATE = "pending"` ✅

## Testing Translations

### Change Language

To test German translations:

1. Open browser DevTools console
2. Run: `localStorage.setItem('language', 'de')`
3. Reload the extension

To switch back to English:
```javascript
localStorage.setItem('language', 'en')
```

### Verify Type Safety

Run the type checker:
```bash
npm run typecheck
```

If you see errors like:
```
Argument of type '"Your key"' is not assignable to parameter of type 'keyof i18nMessages'
```

This means you forgot to add the key to the `i18nMessages` interface.

## Troubleshooting

### "Property does not exist on type 'i18nMessages'"

**Cause**: Translation key not declared in the interface.

**Fix**: Add the key to the `i18nMessages` interface in `src/utils/i18n-extension.ts`.

### Missing translation at runtime

**Cause**: Forgot to import `i18n-extension.js`.

**Fix**: Add import at the top of your file:
```typescript
import "../utils/i18n-extension.js";
```

### Translation not showing in German

**Cause**: Key exists in `en` but not in `de`.

**Fix**: Add the German translation to `lazyboyTranslations.de`.

## Contributing Translations

When adding new features:

1. Add English strings first (development language)
2. Add type declarations for all new keys
3. Add German translations (use DeepL or similar for accuracy)
4. Test both languages before submitting PR
5. Run `npm run typecheck` to verify type safety

For German translations, consider:
- Formal vs. informal tone (use formal "Sie" for consistency)
- Technical terms may stay in English (e.g., "Tab", "Skill")
- Compound nouns in German should be properly combined
